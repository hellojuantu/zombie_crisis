# Zombie Crisis Agent Rules

本文件是后续 Codex/Claude/其他 agent 修改本仓库时必须遵守的工程规则。目标是把游戏继续往“多人、流畅、恐怖、有持续压力的撤离生存游戏”推进，而不是回到简单刷怪原型。

## 项目方向

- 这是个人游戏项目，不需要 legacy 兼容。旧玩法、旧 UI、旧字段、旧空方法如果已经不用，应删除，不要保留兼容层。
- 当前核心玩法是恐怖设施撤离：探索设施楼层、收集任务物、触发风险事件、找到满足条件的撤离终端、进入整备、继续下一关。
- 不要把游戏改回单纯“站桩打僵尸”或“守基地”。本仓库已移除基地防守方向，新增代码不得重新引入 `base`/避难所兼容字段。
- 每个新玩法元素必须有风险回报。回血、搜仓库、进样本房、撤离读条都应该制造压力或消耗资源，而不是白给奖励。

## 架构边界

- 服务端是权威状态源。玩家位置、碰撞、子弹命中、僵尸 AI、任务物、撤离、背包资源、天赋升级、关卡推进都必须由服务端确认。
- 前端负责输入、预测、插值、渲染、HUD、音效和本地视觉反馈；不要让前端决定真实伤害、真实掉落、真实通关。
- `server_asgi.py` 只做 ASGI/Socket.IO 生命周期和事件入口，具体玩法逻辑放在 `server_game/`。
- `server_game/simulation.py` 已经偏大。新增大功能前，优先考虑低风险拆分：
  - `snapshots.py`：快照打包、AOI 裁剪、tuple 编码。
  - `world.py`：地图生成、房间、障碍、A\*、出生点。
  - `missions.py`：任务物、撤离点、整备、路线奖励、关卡推进。
- `static/js/game/app.js` 不应继续无限膨胀。新增网络事件较多时，优先抽 `net.js` 或小 reducer，而不是把所有 socket handler 堆进主循环文件。
- 渲染和服务端逻辑必须分开。服务端不得依赖 DOM/Canvas，前端不得复制服务端规则来决定真实结果。

## 多人同步与性能

- Socket.IO 只使用 websocket transport，避免局域网多人退化到轮询。
- 服务端固定 tick，快照按较低频率发送；高频输入、AOI 快照、附近事件定向发送是默认策略。
- 新增同步字段时必须同时更新：
  - 服务端 tuple/快照打包。
  - `static/js/protocol.js` 解码。
  - 对应协议测试。
- `_player_tuple` 这类 positional tuple 是高风险区域。改索引必须加或更新测试，避免静默错位。
- 玩家断线、超时、重连、整备中离开，都必须清理服务端状态，不能留下 ghost sid。
- 任何会影响多人状态的按钮或 UI 文案，必须由服务端状态驱动；不要只在 UI 上做“看起来准备好了”。

## 玩法状态机

- 关卡状态必须清晰：探索中 -> 条件满足 -> 撤离读条 -> 整备 -> 下一关。
- 整备期必须冻结当前关卡战斗，清理子弹，保护玩家。多人整备必须等待所有存活玩家 ready 后再推进。
- 背包打开时只暂停该玩家，不应让撤离读条继续积累。
- 死亡、重生、背包暂停、整备、断线、重启游戏这些状态不能互相打架；改动时优先补状态转换测试。
- 进入下一关不要自动瞬间清空让玩家看不清。整备界面应保留路线奖励、档案提示和升级选择，由玩家主动进入下一关。

## 迷雾与感染源

- 迷雾不是纯视觉效果。迷雾、感染源、样本房共振、病房止血、机房供电、仓库警报、撤离读条，都应该能制造额外感染体压力。
- 主波次预算和感染源压力池是两套概念。主波次打完后，只要感染源仍有预算，迷雾/危险事件仍可刷少量感染体，避免地图空场。
- UI 需要让玩家知道“感染源仍在活动”或“迷雾带来了额外感染体”，否则玩家会误以为是 bug。
- 不要用无限刷怪粗暴拖时间。感染源压力应有限、有提示、有节奏，服务于恐怖和风险选择。

## 设施与地图

- 地图不要只是正方形迷宫。设施楼层应有病房、机房、样本库、仓库、设备、血迹、警示灯、污染区等可识别区域。
- 房间进入必须有反馈：状态提示、音效/视觉、任务影响、奖励或风险。
- 样本房不能只是装饰。进入样本房应触发共振、提高样本掉落，并带来感染体压力。
- 仓库不应无限白给补给。搜到奖励后要消耗房间状态，并触发警报或压力。
- 僵尸必须能绕障碍找玩家；新增障碍/房间布局后要验证 AI 不会卡在墙外。

## 前端体验

- HUD 保持克制。常用信息只保留生命、武器/弹药、轻量 Ping、目标和撤离条件；调试指标不要常驻玩家 UI。
- 背包用一个小图标入口，点开全屏查看全部资源、武器、撤离条件和当前目标。
- 首页和教学要恐怖但不密集。首局教学只出现一次，核心只教 WASD、瞄准、攻击/贴脸自动刀、开始游戏。
- 手机端提示用户用电脑键鼠玩，不要为了移动端牺牲当前键鼠体验。
- 前端改动后必须提高静态资源版本号，避免浏览器继续加载旧 module，例如 `app.js?v=38`。
- 改 UI 后用浏览器验证实际加载版本和控制台错误，不要只跑静态检查。

## 测试要求

- 服务端改动至少跑：
  - `python3 -m py_compile server_asgi.py server_game/*.py`
  - `python3 -m unittest discover -s tests -v`
- 前端改动至少跑：
  - `node --check static/js/game/app.js`
  - `node --check static/js/game/ui.js`
  - `node --check static/js/game/render.js`
  - `node tests/test_protocol.js`
  - `node tests/test_prediction.js`
  - `node tests/test_interpolation.js`
  - `node tests/test_camera.js`
  - `node tests/test_timing.js`
  - `node tests/test_netcode.js`
- 以下场景属于高风险，改到必须有测试：
  - 多人整备 ready gate。
  - 整备中断线清理。
  - 重启游戏时清理整备和暂停状态。
  - 迷雾在主波次预算为 0 时仍能使用感染源压力池。
  - 冲刺不能穿墙。
  - 武器切换、换弹、备用弹、空弹贴脸自动刀。
  - 房间奖励只触发一次，危险房间重复进入仍有合理反馈。
  - 协议 tuple 索引变更。

## 清理规则

- 不要保留“以后可能有用”的空方法、空字段、旧 UI 分支、旧 payload 字段。
- 如果删除玩法方向，例如基地防守，就要同时删服务端常量、快照字段、前端状态、渲染函数和测试断言。
- 不要添加只为了兼容旧浏览器缓存/旧协议/旧玩法的分支。这个项目不需要 legacy compatibility。
- 若 Claude 或其他审计指出 blocker，应优先修具体 bug，再做大拆分。

## 提交前检查

- 确认 `git status --short` 中新增文件都是有意加入。
- 确认服务端已用最新代码重启。
- 确认浏览器实际加载最新 `?v=` 资源。
- 确认控制台没有当前版本脚本错误。
- 最后给用户说明：改了什么、测试跑了什么、服务地址是什么。
